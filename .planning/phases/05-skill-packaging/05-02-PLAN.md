This file is a decision-tree reference. The calling agent follows these steps mechanically when it receives an error code. Write every recovery path as numbered steps — no prose descriptions without steps (D-14).

**Opening:**
One sentence: When any `node outlook.js` invocation returns `{"status":"error"}`, read `error.code` and follow the matching decision tree below.

Include a code example of how to extract the error code:
```bash
RESULT=$(node outlook.js search "is:unread")
CODE=$(echo "$RESULT" | node -e "const d=JSON.parse(require('fs').readFileSync('/dev/stdin','utf8')); process.stdout.write(d.error ? d.error.code : '')")
```

**Error code sections — one H2 per code:**

**## SESSION_INVALID**

When you see: `{"status":"error","error":{"code":"SESSION_INVALID","message":"..."}}`

This means: The session expired during operation. The skill successfully started but the session became invalid mid-run.

Steps (D-15 — session expired after prior successful invocation):
1. Run: `node outlook.js auth`
2. If a browser window opens: Prompt the user to complete the login (including MFA if required). Wait for the user to confirm they have logged in.
3. Re-run: `node outlook.js auth` (headless check — should return `{"status":"ok"}` immediately if login succeeded)
4. Retry the original operation with the exact same arguments
5. If SESSION_INVALID recurs: Surface the raw `error.message` to the user and ask them to check their session manually

**## AUTH_REQUIRED**

When you see: `{"status":"error","error":{"code":"AUTH_REQUIRED","message":"..."}}`

This means: The session was absent or invalid at startup. Treat identically to SESSION_INVALID.

Steps:
1. Run: `node outlook.js auth`
2. If a browser window opens: Prompt the user to complete the login. Wait for confirmation.
3. Retry the original operation
4. If AUTH_REQUIRED recurs: Same as SESSION_INVALID step 5

Note: In the current codebase, SESSION_INVALID is the primary operational code for expired sessions. AUTH_REQUIRED is listed in requirements but rarely emitted in production paths.

**## INVALID_ARGS**

When you see: `{"status":"error","error":{"code":"INVALID_ARGS","message":"..."}}`

This means: A required argument or environment variable is missing or malformed. This is a programming error or configuration problem — do NOT retry.

Steps:
1. Read `error.message` — it names the specific missing argument or env var
2. If message contains "Missing required env var": Check `.env` file at the project root. Ensure OUTLOOK_BASE_URL, OUTLOOK_BROWSER_PATH, and OUTLOOK_BROWSER_PROFILE are all set. Do NOT retry until fixed.
3. If message contains "Unknown subcommand" or "Missing": Fix the command syntax. Valid subcommands: auth, search, read, digest, tune. `search` requires a query string. `read` requires an id argument.
4. Do NOT surface "please retry" to the user — this is always a fixable configuration or argument error

**## OPERATION_FAILED**

When you see: `{"status":"error","error":{"code":"OPERATION_FAILED","message":"..."}}`

This means: The browser operation failed — timeout, empty snapshot, element not found, or batch command error.

Steps:
1. Retry the exact same operation once (transient browser failures are common)
2. If the retry succeeds: Proceed normally
3. If OPERATION_FAILED recurs:
   a. Surface the raw `error.message` to the user
   b. Suggest the user check if Outlook is accessible in their browser manually
   c. If the operation was `auth` and OPERATION_FAILED: The browser may have failed to launch. Check that OUTLOOK_BROWSER_PATH points to a valid managed Chrome/Edge binary.
4. Do NOT retry more than once automatically — persistent OPERATION_FAILED usually indicates a UI change or environment problem

**## MESSAGE_NOT_FOUND**

When you see: `{"status":"error","error":{"code":"MESSAGE_NOT_FOUND","message":"..."}}`

This means: The `read` subcommand searched for the email by its ConversationId but did not find it in the result list. This happens when the email does not appear in the search results `read` uses internally.

Steps:
1. Retry `read` with the `--query` flag from the original `search` that produced the id:
   `node outlook.js read <id> --query "<original search query>"`
2. If read succeeds: Proceed normally
3. If MESSAGE_NOT_FOUND recurs even with --query:
   a. Confirm the `id` value is from a recent `search` or `digest` result (it should start with `AAQ` — ConversationId format)
   b. The email may have been deleted, archived, or moved since the search was performed
   c. Inform the user and suggest running a new `search` to get a fresh id

**Closing:**

Add a brief note: `auth` is idempotent and safe to run at any time. If uncertain about session state before a series of operations, run `node outlook.js auth` first — it returns immediately if the session is already valid.

This file teaches the calling agent the digest scoring algorithm so it can explain digest results to users in natural language. Read scoring.json.example and lib/digest.js before writing to ensure all values are authoritative (D-16).

**Structure:**

**Opening (2 sentences):**
The `digest` subcommand scores each today's inbox message and returns results sorted by `importance_score` descending. This file documents the scoring algorithm, all signal values, and natural language templates for explaining results to users.

**Section 1: Score Scale**
Heading: `## Score Scale`

| Range | Meaning |
|-------|---------|
| 0 | No signals fired — email is read AND sender is not a channel AND no urgency keywords |
| 1–20 | Low importance — keyword signal only, or channel email with keywords |
| 21–40 | Medium — unread channel email, or keyword-heavy read email |
| 41–60 | High — unread human email, no keywords |
| 61–80 | Very high — unread human + keywords, or unread human + high importance |
| 81–100 | Critical — high importance + unread + keywords |

Note: Theoretical maximum is 100 (unread_human 40 + high_importance 40 + keyword_max 20).

**Section 2: Default Scoring Weights**
Heading: `## Default Scoring Weights`

These are the defaults from `scoring.json.example`. If `scoring.json` exists at the project root, it overrides these defaults. (D-16)

| Weight Key | Default Value | When Applied |
|------------|---------------|-------------|
| `unread_human` | 40 | Email is unread AND sender is detected as a human (name with comma, or 1–2 word non-channel name) AND sender does not match bulk patterns |
| `unread_channel` | 20 | Email is unread AND sender is detected as a channel/group AND not bulk |
| `high_importance_prefix` | 40 | Outlook marked the email with "Important " accessibility prefix |
| `keyword_per_match` | 5 | Each urgency keyword matched in from+subject+preview field |
| `keyword_max` | 20 | Maximum total from keyword matches (capped even if many keywords match) |

Note: The `bulk` suppression removes the unread bonus entirely — if a sender matches bulk patterns, neither `unread_human` nor `unread_channel` is added, regardless of read status.

**Section 3: All importance_signals Values**
Heading: `## importance_signals Values`

The `importance_signals` array in each digest result tells you exactly which signals fired. (D-17)

| Signal Value | Plain-English Meaning | Points |
|-------------|----------------------|--------|
| `"unread"` | Email is unread from a human sender | +40 (default) |
| `"unread:channel"` | Email is unread from a channel, team, or group address | +20 (default) |
| `"high_importance"` | Outlook flagged this email as high importance (! icon) | +40 (default) |
| `"keyword:<term>"` | The word or phrase `<term>` appeared in the sender, subject, or preview | +5 per match, max +20 (default) |
| `"bulk"` | Email is unread but the unread bonus was suppressed — sender is a newsletter, digest, or no-reply address | 0 (suppression) |

Note on `keyword:<term>` format: The full signal string includes the matched keyword after the colon. Examples: `"keyword:urgent"`, `"keyword:eod"`, `"keyword:action required"`. The term matches the exact entry in `urgency_keywords`.

**Section 4: Default Urgency Keywords**
Heading: `## Default Urgency Keywords`

These keywords are checked against the combined `from + subject + preview` text (case-insensitive). From `scoring.json.example`:

```
urgent, asap, action required, immediate, deadline,
due today, overdue, eod, eow, critical, expires, time-sensitive,
itinerary, booking confirmation, reservation, check-in, travel alert
```

Add or remove keywords by editing `scoring.json` (copy from `scoring.json.example`).

**Section 5: Human vs Channel Sender Detection**
Heading: `## Sender Classification`

The skill classifies senders to decide which unread weight applies. This logic runs on the `from` field (which contains the combined sender+subject text from the list view).

A sender is classified as **human** if:
- The name contains a comma in the first 40 characters (e.g. "User, Alpha" → human)
- OR the name is 1–2 words with no channel indicator word

A sender is classified as **channel/group** if:
- The name is 3+ words
- OR the name contains any of these channel indicator words: `group`, `team`, `circle`, `community`, `center`, `online`, `network`, `office`, `alliance`, `program`, `service`, `digest`, `erg`, `management`

A sender is classified as **bulk** (unread bonus suppressed) if the name or combined field matches any entry in `bulk_patterns` from `scoring.json.example`:
`no reply`, `noreply`, `no-reply`, `donotreply`, `do not reply`, `bulk email`, `email digest`, `end user digest`, `newsletter`, `unsubscribe`, `started a new conversation`, `new posts from`

**Section 6: Natural Language Explanation Templates (D-18)**
Heading: `## Natural Language Explanation Templates`

Use these templates when explaining a digest result to the user. Replace the bracketed values with actual data from the result.

```
# Single signal
"This email from [from] scored [importance_score] because it is unread."
"This email from [from] scored [importance_score] because it is marked high importance."
"This email from [from] scored [importance_score] — matched urgency keyword: [keyword term]."

# Multiple signals
"This email from [from] scored [importance_score] because it is unread (+40) and marked high importance (+40)."
"This email from [from] scored [importance_score]: unread (+40), matched keywords [keyword signals joined by comma] (+[total keyword points])."

# Bulk suppression (D-19)
"This email from [from] scored [importance_score]. It is unread, but the unread bonus was suppressed because this appears to be a newsletter or automated email."

# Channel email
"This email from [from] is from a team or group address. It scored [importance_score] as an unread channel email."

# Zero score
"This email from [from] scored 0 — it has been read and no urgency signals were detected."
```

**Section 7: Calibrating Scores (D-20)**
Heading: `## Calibrating Scores`

If the default scores don't reflect your inbox priorities:
1. Copy `scoring.json.example` to `scoring.json` in the project root
2. Edit `scoring.json` to adjust weights, keywords, or bulk patterns
3. Run `node outlook.js tune` to evaluate the new configuration against a sample of your inbox
4. Run `node outlook.js tune --save` to persist the effective configuration back to `scoring.json`

The `tune` command samples the inbox across multiple queries, grades the distribution into 5 tiers (Tier 0 = low noise / automated through Tier 4 = critical), and returns a `verdict` of `PASS`, `WARN`, or `FAIL`. A `FAIL` verdict means more than 40% of sampled messages land in Tier 3+4 — which usually means weights are too aggressive or bulk patterns are missing for your common senders.
