**Document sections (in order):**

**1. Introduction paragraph:**
"The `calendar`, `calendar-read`, and `calendar-search` subcommands provide read-only access to the user's Outlook calendar. This file documents the JSON schemas for all three subcommands, the meaning of each field, and known limitations."

**2. `calendar` Listing Schema (D-07):**

Table documenting each field:

| Field | Type | Description |
|-------|------|-------------|
| `id` | string or null | Stable event identifier. Pass to `calendar-read` unchanged. May be null if the calendar view does not expose a stable DOM attribute (see ID Limitations below). |
| `subject` | string | Event title/subject |
| `organizer` | string | Name of the event organizer |
| `start_time` | string | ISO 8601 UTC datetime (e.g., `"2026-04-14T13:00:00.000Z"`) |
| `end_time` | string | ISO 8601 UTC datetime |
| `duration_minutes` | number | Duration in minutes. 1440 for all-day events. |
| `location` | string or null | Meeting location text, or null if not specified |
| `is_online_meeting` | boolean | True if the event has a Teams/Zoom meeting link |
| `is_all_day` | boolean | True for all-day events |
| `is_recurring` | boolean | True if this is an instance of a recurring event series |
| `response_status` | string | One of: `"accepted"`, `"tentative"`, `"none"`, `"unknown"` |

**3. `calendar-read` Detail Schema (D-10):**

All fields from D-07 plus:

| Field | Type | Description |
|-------|------|-------------|
| `meeting_link` | string or null | Teams or Zoom URL extracted from event body. Null if no meeting link found. |
| `attendees` | array | List of `{ "name": "string", "email": "string or null", "response": "accepted/tentative/declined/none" }` objects. Empty array if no attendees visible. |
| `body_text` | string | Event agenda/notes as plain text. May contain meeting join instructions, agenda items, etc. |

**4. `response_status` Values:**

Table with: `"accepted"` = User accepted the meeting invitation, `"tentative"` = User tentatively accepted, `"none"` = User has not responded to the invitation, `"unknown"` = Response status could not be determined from the calendar view.

Explain: "In the `calendar` listing, `response_status` may be `"unknown"` if the status is not visible in the event row accessible name. Use `calendar-read` to get the definitive response status from the detail view."

**5. `--days` Flag Behavior (D-05):**

- `calendar --days 7` (default): lists events from today through 7 calendar days ahead
- `calendar --days 1`: today's events only
- `calendar --days 30`: next 30 days (may require scroll-accumulate for busy calendars)
- Events are sorted chronologically by `start_time` ascending
- All-day events for a given day appear alongside timed events for that day

**6. Event ID Limitations (D-08):**

Document the actual ID situation discovered during ARIA exploration. If stable IDs are available, document the attribute name. If using composite key fallback, explain that event IDs may be composite keys (stringified `{subject, start_time}` objects) and that two events with identical subject and start_time would share the same ID. Always pass the ID to `calendar-read` exactly as received.

**7. `calendar-search` Query Operators:**

Document which KQL-style operators work for calendar search based on implementation findings. Reference `references/kql-syntax.md` for email operators. Note differences. Cover: `subject:`, `before:` / `after:`, free-text, and which email operators do NOT apply to calendar.

**8. Natural Language Explanation Templates:**

Templates for: single event, all-day event, online meeting with join link, recurring event, calendar listing summary.
