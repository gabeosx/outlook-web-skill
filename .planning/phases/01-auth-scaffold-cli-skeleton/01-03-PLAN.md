Set environment variables (adjust paths to your actual Chrome installation and profile):

```bash
export OUTLOOK_BASE_URL="https://mail.example.com"
export OUTLOOK_BROWSER_PATH="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
export OUTLOOK_BROWSER_PROFILE="$HOME/Library/Application Support/Google/Chrome/Default"
```

To find your correct Chrome profile path: Open Chrome, navigate to chrome://version, look for "Profile Path". Use the full path shown (e.g., `/Users/you/Library/Application Support/Google/Chrome/Default`).

Step 1 — First invocation (should open browser window):
```bash
node outlook.js auth
```
Expected: A Chrome window opens at your Outlook URL. If not already logged in, complete MFA/SSO interactively. The skill waits up to 5 minutes.

Step 2 — Verify session was saved:
```bash
ls -la ~/.agent-browser/sessions/
```
Expected: A file named `outlook-skill-default.json` or similar exists.

Step 3 — Second invocation (should NOT open browser window):
```bash
node outlook.js auth
```
Expected: No browser window opens. stdout prints: `{"operation":"auth","status":"ok","results":null,"error":null}`

Step 4 — Verify stdout is valid JSON:
```bash
node outlook.js auth | node -e "const d=require('fs').readFileSync('/dev/stdin','utf8');JSON.parse(d);console.log('valid JSON')"
```

If bot detection blocks the browser (Conditional Access error instead of login form), try:
- Using Edge: `export OUTLOOK_BROWSER_PATH="/Applications/Microsoft Edge.app/Contents/MacOS/Microsoft Edge"`
- The --auto-connect fallback: open Chrome with Outlook loaded, then in a terminal: `agent-browser --auto-connect --session-name outlook-skill state save`

To capture debug output: `node outlook.js auth > /tmp/auth-stdout.txt 2>/tmp/auth-stderr.txt`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User's live Outlook tenant → skill auth flow | Real Microsoft Entra authentication; real session tokens |
| ~/.agent-browser/sessions/ → disk | Session file contains real authentication cookies and localStorage tokens |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | ~/.agent-browser/sessions/ session file contains real auth tokens | accept | Session files are in the user's home directory with default OS permissions. This is equivalent to Chrome's own cookie store. The skill does not log, transmit, or process the session file contents — agent-browser manages it internally. Do not commit ~/.agent-browser/ to git. |
| T-03-02 | Information Disclosure | /tmp/auth-stdout.txt and /tmp/auth-stderr.txt (debug files) | accept | Debug files are created only if the user explicitly runs the debug capture command. They are in /tmp/ which is ephemeral. The skill itself does not create these files. |
| T-03-03 | Spoofing | Conditional Access error page appears at a non-login URL | accept | If Entra routes the Conditional Access error to a non-login domain URL, `isLoginUrl()` would falsely return "valid session". This is an edge case requiring RESEARCH.md Pitfall 2 fallback path. Document if observed. |
</threat_model>
