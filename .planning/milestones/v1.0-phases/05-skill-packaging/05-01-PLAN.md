**5. Subcommands**
Heading: `## Subcommands`

For each subcommand, include: syntax, description, example success response (annotated JSON — D-06), and critical calling-agent notes.

**5a. auth**
Syntax: `node outlook.js auth`
Description: Checks session validity. If session is valid, returns immediately without opening a browser (idempotent). If session is invalid or expired, opens a headed Chrome window for the user to complete login. Run before any other subcommand if uncertain about session state. Safe to run as a pre-flight check.
Success response:
```json
{"operation": "auth", "status": "ok", "results": null, "error": null}
```
Note: If auth itself fails to launch the browser, it returns OPERATION_FAILED (not AUTH_REQUIRED).

**5b. search**
Syntax: `node outlook.js search "<kql-query>" [--limit <n>]`
- `<kql-query>`: KQL query string (required). Read references/kql-syntax.md before constructing queries.
- `--limit <n>`: max results to return (default: 20)
Success response (annotated JSON):
```json
{
  "operation": "search",
  "status": "ok",
  "results": [
    {
      "id": "AAQkADM...",       // ConversationId — pass to 'read' unchanged
      "from": "Smith, Alice",  // WARNING: contains combined sender+subject text from list view
      "subject": "",           // ALWAYS empty string — subject only available via 'read'
      "date": "Mon 4/7/2026",
      "preview": "Here is the document you requested...",
      "is_read": false,
      "is_flagged": null,      // ALWAYS null — not determinable from list view
      "to": null               // ALWAYS null — only visible in reading pane
    }
  ],
  "count": 1,
  "error": null
}
```
Critical notes:
- `subject` is ALWAYS `""`. The combined sender+subject text is in `from`. To get the actual subject line, pass the `id` to `read`.
- `id` is a ConversationId (`data-convid`). Pass it directly to `read` without modification.
- Zero results returns `{"status":"ok","results":[],"count":0}` — not an error.

**5c. read**
Syntax: `node outlook.js read <id> [--query <search-query>]`
- `<id>`: ConversationId from a prior search or digest result (required)
- `--query <search-query>`: Optional. Pass the KQL query from the preceding search so `read` can locate the email in context. Without this flag, `read` uses a broad space-search; a specific email from weeks ago may not appear.

Success response (annotated JSON):
```json
{
  "operation": "read",
  "status": "ok",
  "results": {
    "subject": "Q1 Budget Review",
    "from": "Smith, Alice <alice@example.com>",
    "to": "You; Bob Jones",
    "cc": "Manager, Jane",          // null when no CC recipients
    "date": "Mon 4/7/2026 9:15 AM",
    "body_text": "Hi team,\nPlease review the attached...",
    "has_attachments": true,
    "attachment_names": ["Q1_Budget.xlsx"]  // empty array when no attachments
  },
  "error": null
}
```
Note: `results` is a single object (not an array) for `read`. `subject` is populated here (unlike search/digest).

**5d. digest**
Syntax: `node outlook.js digest`
Description: Fetches today's inbox messages and returns them sorted by importance score descending. Reads only the "Today" group from the inbox. See references/digest-signals.md for scoring explanation.

Success response (annotated JSON):
```json
{
  "operation": "digest",
  "status": "ok",
  "results": [
    {
      "id": "AAQkADM...",
      "from": "Smith, Alice",    // same combined sender+subject as search
      "subject": "",             // ALWAYS empty string
      "date": "4/12/2026",
      "preview": "Please review before EOD...",
      "is_read": false,
      "is_flagged": null,
      "importance_score": 85,
      "importance_signals": ["unread", "high_importance", "keyword:eod"]
    }
  ],
  "count": 1,
  "error": null
}
```
Note: Empty results (`[]`) means no "Today" group was found in the current inbox view — not necessarily an empty inbox. Try `node outlook.js search "is:unread"` as a fallback.

**5e. tune**
Syntax: `node outlook.js tune [--save]`
Description: Samples the inbox across multiple queries and grades the current scoring configuration. Returns tier distribution (Tier 0 = low noise through Tier 4 = critical). `--save` writes the effective configuration to `scoring.json`. Grade report is written to stderr; the JSON envelope goes to stdout. Use when the digest rankings seem miscalibrated. Verdict values: `PASS` (Tier 3+4 ≤ 25%), `WARN` (26–40%), `FAIL` (>40%).
