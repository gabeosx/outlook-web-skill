**Capture sections (each prefixed with an `echo "=== CAPTURE N: description ==="` separator):**

1. **Inbox view snapshot** -- `agent-browser --auto-connect open "${BASE_URL}/mail/inbox"` then `agent-browser wait 3000` then `agent-browser snapshot -i`. This captures the message list container (`listbox "Message list"`), individual message rows (`option`), reading pane (`main "Reading Pane"`), search box (`combobox`), navigation pane, and Focused Inbox tab presence/absence.

2. **Current URL after inbox load** -- `agent-browser get url`. Confirms landing URL pattern.

3. **Focused Inbox tab detection** -- `echo` a marker line stating: "Look for button 'Focused' or button 'Other' in the snapshot output above. If absent, Focused Inbox is disabled on this account."

4. **Data attribute probe** -- Use `agent-browser eval` with `--stdin` and a heredoc to run:
```javascript
JSON.stringify(
  Array.from(document.querySelectorAll('[role="option"]'))
    .slice(0, 3)
    .map(el => ({
      dataset: Object.keys(el.dataset),
      id: el.id,
      ariaSelected: el.getAttribute('aria-selected')
    }))
)
```
This resolves the open question about `data-convid` or other data attributes on message rows.

5. **Search box activation** -- Re-snapshot after clicking the search combobox. Use `agent-browser find role combobox click --name "Search for email"` (semantic locator, not hardcoded ref). Then `agent-browser wait 1000` then `agent-browser snapshot -i`. This captures the expanded search state with companion buttons.

6. **Search results capture** -- `agent-browser find role combobox fill "from:test" --name "Search for email"` then `agent-browser press Enter` then `agent-browser wait 5000` then `agent-browser get url` then `agent-browser snapshot -i`. This resolves whether search results use the same `listbox "Message list"` container or a different structure.

7. **Open individual message from search results** -- After the search snapshot, identify the first option in the listbox and click it (use `agent-browser find role option click` or the first `@eN` from the search results snapshot -- add a comment noting the ref must be read from the snapshot output). Then `agent-browser wait 2000` then `agent-browser get url` (captures the `/mail/inbox/id/<ItemID>` URL pattern with a concrete ID value) then `agent-browser snapshot -i` (captures reading pane with message body structure).

8. **Attachment search** -- `agent-browser find role combobox click --name "Search for email"` then `agent-browser wait 500` then `agent-browser find role combobox fill "has:attachment" --name "Search for email"` then `agent-browser press Enter` then `agent-browser wait 5000` then `agent-browser snapshot -i`. Then open the first result and snapshot the reading pane. This resolves the attachment indicator ARIA structure (currently `unresolved` in research).

9. **Cleanup** -- `agent-browser close`.

**Important implementation notes:**
- Between capture sections 5-8, the script may need to re-find the search combobox after each search because refs invalidate on page changes. Use semantic locators (`find role combobox --name "Search for email"`) not hardcoded `@eN` refs.
- Add a comment before section 7 saying "NOTE: Read the ref from section 6 snapshot output. Replace @eXX with the first option ref." since this step requires human reading of intermediate output.
- Do NOT use `wait --load networkidle` anywhere (Outlook has persistent background network activity that causes hangs).
- Use `agent-browser wait 3000` or `agent-browser wait 5000` for SPA load waits.
- Make the script executable: the executor should run `chmod +x outlook-capture.sh` after creation.

The script should be designed so that if any step fails, the user can still get partial output from stdout. Use `set -euo pipefail` but add `|| echo "CAPTURE FAILED: [section name]"` to each critical command so partial results are preserved.

| Boundary | Description |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-01 | I (Information Disclosure) | outlook-capture.sh output | accept | The capture script output (`/tmp/outlook-snapshots.txt`) contains real email metadata (sender names, subjects, preview text). This is inherent to the exploration task. The file is created in `/tmp/` (auto-cleaned by OS), the capture script is disposable (deleted after use per D-01), and the Evidence section of `outlook-ui.md` will contain only accessibility tree metadata -- not full email bodies. | T-00-02 | I (Information Disclosure) | references/outlook-ui.md Evidence section | mitigate | Raw snapshot dumps in the Evidence section will contain real sender names and subject lines. Mitigation: Task 3 action explicitly states to NOT include email body content beyond what appears in the accessibility tree. No passwords, tokens, or authentication details are included. The file is committed to a local repo -- if the repo is ever shared publicly, the user should scrub the Evidence section. |
| T-00-03 | T (Tampering) | --auto-connect session import | accept | `--auto-connect` connects to the user's running Chrome instance. This is the intended behavior for Phase 0 exploration. The connection is local (localhost CDP), not exposed to the network. No persistent session state is created (per D-05). |
| T-00-04 | E (Elevation of Privilege) | capture script commands | accept | The capture script uses `snapshot`, `click`, `fill`, `press`, `wait`, `get`, `eval`, and `close` commands. No Action Policy is applied (this is exploration, not production). The script interacts only with the user's own Outlook inbox. The `eval` command probes DOM data attributes (read-only).
